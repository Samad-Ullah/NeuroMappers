## 3. What is actually shipped on OSF

Contrary to how the paper frames Lead-DBS outputs, the OSF download contains **raw native-space NIfTI images only**.

```
data/raw/LeadTutor/
├── dataset_description.json          # BIDS 1.6.0 / Lead-DBS v3.1
├── rawdata/
│   └── sub-XXXXX/
│       ├── ses-preop/anat/           # pre-operative MRI (T1w, often T2w, rarely FLAIR/SWI)
│       └── ses-postop/anat/          # post-operative CT
├── derivatives/
│   └── leaddbs/sub-XXXXX/prefs/      # Lead-DBS UI prefs only (*.json, *_uiprefs.mat)
└── sourcedata/                       # empty
```

**31 NIfTI files total across the 10 subjects.**

What is **not** included (and therefore must be produced by our pipeline):
- No pre-registered CT↔MRI volumes
- No MNI-normalized images
- No electrode reconstructions
- **No expert ground-truth electrode coordinates** (needs to be requested from Prof. Cunha for Stage 5 evaluation)

## 5. Per-file inspection table — `results/sprint1_data_inspection.csv`

Generated by `notebooks/exploration.ipynb` (function `inspect_many` in `src/load_and_inspect.py`). One row per NIfTI file. Columns:

| Column | Meaning | Why it matters |
|--------|---------|----------------|
| **path** | Absolute path to the NIfTI file | Identifies subject, session (preop/postop), and modality |
| **modality_guess** | `T1w` / `T2w` / `FLAIR` / `SWI` / `CT` / `unknown`, inferred from filename | Separates MRI modalities from CT for pipeline branching |
| **format** | `NIfTI-1` or `NIfTI-2` | All 31 files are NIfTI-1; confirms standard loading works |
| **shape** | Voxel grid dimensions `(X, Y, Z)` | MRI and CT shapes differ; affects memory and resampling |
| **voxel_spacing_mm** | Physical voxel size in mm along each axis | Drives the **resampling** step before registration |
| **orientation** | 3-letter axis code (e.g., `('L','A','S')`) | Drives the **reorientation** step before fusion |
| **dtype** | Numeric type of voxel data (`float64`, `int16`, …) | Affects memory footprint and NaN possibility |
| **min_intensity** | Minimum voxel value (NaN-safe) | CT ~ −1024 HU (air); MRI ~ 0 |
| **max_intensity** | Maximum voxel value (NaN-safe) | CT bone/electrodes 3000+ HU; MRI varies |
| **nan_count** | Number of `NaN` voxels | Quality flag — see §7 for sub-39468 |

## 6. Data heterogeneity (findings that drive pipeline design)

**Orientations — 4 distinct ones across 31 files:**
- `(L, A, S)` — majority
- `(R, A, S)`
- `(P, S, R)`
- `(R, S, A)`

→ Reorientation to a common frame (e.g., RAS) is **mandatory** before registration.

**Voxel spacing:**
- **CT:** ~0.4–0.5 mm in-plane × 0.5–2 mm slice thickness (anisotropic, fine in-plane)
- **T1w:** mostly isotropic ~1 mm
- **T2w:** anisotropic, slice thickness 1.2–3 mm

→ Resampling to a common grid is required before multimodal fusion.

**File format:** all 31 files are NIfTI-1; all load as `float64` via nibabel `dataobj`.

## 7. Data-quality issues flagged for Prof. Cunha

1. **sub-39468 has SWI, not FLAIR** — the paper (Table 1) lists this subject as having FLAIR, but the file is actually Susceptibility-Weighted Imaging. Filename and header both confirm SWI.
2. **sub-39468 post-op CT has ~7,452,385 NaN voxels (~21 % of the volume)** — the CT is partially corrupted / contains large regions of undefined data. Cannot be registered or resampled without explicit NaN handling. Needs a decision on whether to include or exclude this subject.
